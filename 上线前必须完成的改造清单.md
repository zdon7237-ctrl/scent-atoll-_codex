# 香水店微信小程序上线前必须完成的改造清单

## 当前结论

当前版本更接近“高完成度业务原型/演示版”，还不适合直接用于正式运营。  
核心原因不是页面多少，而是交易闭环仍停留在客户端本地：支付、钱包、订单、库存、优惠券、会员资产、后台权限都还没有形成可审计、可防刷、可多端同步的正式链路。

---

## 一、上线前必须先完成

下面这些属于首发阻断项，建议全部完成后再考虑正式上线。

### 1. 建立真实用户身份体系

- [x] 接入正式用户标识链路
- [ ] 后端完成 `openid / unionid / session` 绑定
- [ ] 所有用户资产都与用户唯一身份绑定

当前进展：
- 已接入基于微信云开发的 `openid` 用户识别与用户建档。
- 小程序启动时已完成登录初始化，并把当前用户资料同步到前端全局状态。
- `unionid / session` 能力与资产全量绑定仍留到后续阶段。

目标：
让订单、优惠券、钱包、积分、会员等级都归属于真实用户，而不是当前设备。

为什么必须先做：
- 现在的会员数据、消费额、订单、钱包余额都主要依赖本地存储。
- 用户换设备、清缓存、重装小程序后，数据会丢失或失真。

验收标准：
- 同一个用户在两台设备登录后，可看到一致的订单、积分、优惠券、钱包余额。
- 后台能根据用户身份查到完整交易记录。

### 2. 把订单系统迁到服务端

- [x] 服务端创建正式订单表/集合
- [x] 下单动作改为服务端创建订单
- [x] 订单号由服务端生成
- [ ] 订单状态变更由服务端控制

当前进展：
- 已新增云端 `orders` 集合与服务端订单创建/列表/详情读取能力。
- 购物车结算已改为创建云端 `待付款` 订单，不再依赖本地 `myOrders`。
- 支付后的状态推进、发货、收货、售后流转仍待下一阶段接入。

目标：
订单要成为平台的真实业务记录，而不是前端页面上的本地列表。

为什么必须先做：
- 当前订单直接写入本地 storage。
- 发货、收货、售后状态也都可以在客户端自己改。

验收标准：
- 下单后，订单数据可在服务端持久保存。
- 用户删除本地缓存，订单仍可恢复。
- 商家与用户看到的是同一笔订单状态。

### 3. 接入真实支付能力，移除本地“模拟支付成功”

- [ ] 设计正式支付流程
- [ ] 服务端生成预支付参数
- [ ] 小程序侧接入微信支付拉起支付
- [ ] 支付回调由服务端确认订单支付结果

目标：
让“付款成功”来自真实支付结果，而不是前端直接弹 Toast。

为什么必须先做：
- 当前结算逻辑在客户端直接创建订单、加积分、累计消费并提示支付成功。
- 这不具备任何真实收款能力。

验收标准：
- 未支付时不能进入已支付后的订单状态。
- 支付成功只能由支付回调确认。
- 可区分待支付、支付成功、支付失败、关闭订单等状态。

### 4. 钱包/储值功能必须服务端化，首发前建议谨慎上线

- [ ] 若继续保留钱包，需改为服务端账本
- [ ] 充值、消费、退款都要留流水
- [ ] 钱包余额变动要可审计、可追溯
- [ ] 如果短期做不完，建议首发先下线钱包入口

目标：
钱包余额必须是真实资产，不允许前端自增自减。

为什么必须先做：
- 当前钱包充值本质上是本地加数字。
- 这会直接造成资金风险。

验收标准：
- 钱包余额只能通过服务端交易更新。
- 任意一笔余额变化都能在后台追踪来源。
- 前端无法通过本地修改制造余额。

### 5. 建立库存校验与扣减机制

- [ ] 商品库存存放到服务端
- [ ] 加购与下单前做库存校验
- [ ] 支付成功后正式扣减库存
- [ ] 处理超卖、并发下单、订单取消回补库存

目标：
库存从“展示信息”升级为“交易约束条件”。

为什么必须先做：
- 现在库存只显示不拦截，购物车加数量和下单都不校验库存。
- 正式卖货时会直接出现超卖。

验收标准：
- 库存不足时不能下单。
- 并发下单时不会出现负库存。
- 退款/取消流程有明确的回补规则。

### 6. 增加收货地址、联系人、物流字段与履约流程

- [ ] 增加收货人姓名
- [ ] 增加手机号
- [ ] 增加收货地址
- [ ] 增加物流单号、发货时间、物流状态
- [ ] 商家侧补充发货操作界面

目标：
让订单具备真实履约能力，而不是只停留在“已下单”。

为什么必须先做：
- 当前订单结构不含地址、联系人、物流等核心信息。
- 没有这些字段，香水无法完成真实寄送。

验收标准：
- 用户下单前必须提交完整收货信息。
- 商家可录入发货信息，用户可查看物流状态。

### 7. 优惠券改为服务端发放、领取、核销

- [ ] 用户券包迁移到服务端
- [ ] 自动发券由服务端执行
- [ ] 兑换码在服务端校验与核销
- [ ] 下单时服务端校验券是否可用
- [ ] 防止重复领取、重复核销、跨设备重放

目标：
优惠券成为可控资产，而不是本地临时状态。

为什么必须先做：
- 当前自动发券、兑换码校验、已使用标记都发生在客户端本地。
- 清缓存或换设备后存在重领、重放和资产失真风险。

验收标准：
- 每张券的领取、使用、过期都能在服务端查询。
- 同一张券不能被重复核销。
- 同一兑换码具备明确使用规则和风控限制。

### 8. 会员等级、积分、累计消费额迁移到服务端

- [ ] 累计消费额改为服务端累计
- [ ] 会员等级由服务端计算
- [ ] 积分获取与消耗进入服务端账本
- [ ] 修正新用户默认等级问题

目标：
会员权益必须基于真实交易数据计算，且可跨设备同步。

为什么必须先做：
- 当前累计消费额和积分主要在本地维护。
- 现版本新用户默认累计消费额过高，会直接得到高等级会员权益。

验收标准：
- 新用户初始会员等级正确。
- 支付成功后，会员等级和积分按统一规则由服务端更新。
- 积分、等级、累计消费额可在后台核对。

### 9. 下线或严格隔离当前 Dev 后台入口

- [x] 正式环境移除 `后台管理 (Dev)` 入口
- [ ] 或增加管理员身份校验
- [ ] 管理端功能迁移为独立后台

目标：
避免普通用户进入调试页面，避免本地调试逻辑暴露到线上。

为什么必须先做：
- 当前“后台管理”入口在用户中心可直接看到。
- 其中仍包含模拟会员升级、商品折扣开关等开发态能力。

验收标准：
- 普通用户在正式环境看不到 Dev 能力。
- 管理操作必须经过管理员认证与服务端校验。

当前进展：
- 已完成 `devtools` 测试工具页隔离。
- 已从正式业务页移除 Dev 菜单和“模拟发货”等测试入口。
- 正式配置下已通过 `check:release` 守卫检查。

### 10. 修复首页与首屏体验，确保正式可展示

- [ ] 修复首页模板与逻辑不一致问题
- [ ] 明确首页承担的业务角色
- [ ] 补全首屏内容、入口、品牌展示或活动信息

目标：
首屏必须是一个可正式对外展示的页面，而不是残留模板片段。

为什么必须先做：
- 当前首页代码维护商品列表，但首页模板只剩局部价格片段。
- 这会直接影响正式体验与首屏观感。

验收标准：
- 首页可完整打开且内容结构清晰。
- 首页信息与业务定位一致，不存在残片或开发占位内容。

### 11. 完成基础合规与运营必备项

- [ ] 补齐隐私政策与个人信息处理说明
- [ ] 明确收集哪些个人信息、为何收集、如何使用
- [ ] 检查小程序类目与经营资质是否匹配
- [ ] 准备售后说明、配送规则、退款规则

目标：
保证小程序具备正式运营所需的最基本合规材料和用户说明。

为什么必须先做：
- 当前仓库内未体现出完整的隐私、履约、售后等正式运营文案链路。
- 香水属于实物零售，履约和售后说明不能缺位。

验收标准：
- 用户能在小程序内查看隐私和售后规则。
- 提审资料与功能形态保持一致。

### 12. 建立最基本的测试与验收流程

- [x] 增加关键业务流程测试
- [ ] 至少覆盖下单、支付结果、券核销、库存不足、退款/取消
- [x] 建立发布前检查清单

目标：
避免核心交易逻辑只靠人工点点看。

为什么必须先做：
- 当前工程没有成型的脚本化测试或发布脚本。
- 交易链路改造后，如果没有回归测试，后续容易反复出问题。

验收标准：
- 每次准备发布前，都能重复执行一套固定自测流程。
- 核心链路具备明确通过标准。

当前进展：
- 已增加 `devtools` 隔离相关自动测试。
- 已增加 UTF-8 无 BOM 编码回归测试。
- 已增加 `release:prepare` / `check:release` 发布前检查脚本。
- 已增加身份初始化、服务端订单创建、订单列表加载态等关键业务流程测试。
- 真实支付、优惠券核销、库存不足等主链路测试仍未开始。

### 13. 下一轮任务：收货地址 + 订单详情页 + 支付前置结构

- [ ] 在订单模型中加入收货人、手机号、收货地址字段
- [ ] 新增订单详情页并接入 `order_detail` 云函数
- [ ] 为待付款订单补齐支付前置结构与状态流转入口
- [ ] 规划支付成功后累计消费、积分、优惠券、钱包的落账顺序

目标：
让当前“待付款订单”从可创建升级为后续可支付、可履约、可追踪的完整订单。

## 四、最现实的上线策略建议

如果你想更快首发，我建议这样控制范围：

### 方案 A：最稳妥首发

- 保留：商品浏览、品牌页、商品详情、真实下单、真实支付、订单查询、会员中心
- 暂缓：钱包、积分商城、Dev 后台
- 条件：订单、支付、库存、优惠券、身份体系全部服务端化

适用场景：
想尽快把商城真正跑起来，但不想一次把所有营销玩法都做全。

### 方案 B：继续作为演示版/预约版

- 保留当前展示能力
- 不开放真实交易
- 用于内部演示、招商展示、体验验证

适用场景：
短期内还没有时间补完后端交易体系，但想先展示项目能力。
