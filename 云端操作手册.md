## 1. 这份手册是干什么的

这份手册用于指导日常云端数据维护，重点覆盖：

- 商品数据维护（`perfumes`）
- 品牌数据维护（`brands`）
- 优惠券模板维护（`coupons`）
- 节日自动发券与内部兑换码券的配置方式
- 修改云端数据后的验证方法

当前项目的云端主要承担“模板数据源”职责：

- `perfumes`：商品模板
- `brands`：品牌分类模板
- `coupons`：优惠券模板

当前项目**没有**把用户购物车、订单、钱包、用户券包上云，用户状态仍以本地存储为主。

## 3. 优惠券发放规则说明

当前优惠券采用两种发放方式：

### 3.1 自动发券：`issueMode = auto`

适用于：

- 节日券
- 活动通用券
- 用户一上线就应该自动拥有的券

行为：

- 用户启动小程序后，系统会自动把这类券同步到本地券包
- 同一张券按模板 `id` 只会发一次

### 3.2 手动兑换：`issueMode = code`

适用于：

- 内部福利券
- 奖品券
- 线下活动口令券

行为：

- 不会自动发给用户
- 用户必须在优惠券页输入兑换码后才能领取
- 兑换码来自模板字段 `redeemCode`

### 3.3 重要运营约束

**如果未来还要再次发放同类节日活动券，请新建模板 `id`，不要复用旧模板。**

原因：

- 当前版本不会对同一个模板 `id` 重复自动发放
- 如果复用旧模板 `id`，已领过这张券的用户将不会再次收到

## 4. `coupons` 集合字段说明

每条优惠券模板建议包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `id` | number | 是 | 业务主键，必须唯一 |
| `name` | string | 是 | 券名称 |
| `value` | number | 是 | 现金券填金额，折扣券填折扣率，例如 `0.9` |
| `min` | number | 是 | 满多少可用 |
| `type` | string | 是 | `cash` 或 `discount` |
| `startTime` | string | 是 | 开始时间，例如 `2026-03-01 00:00:00` |
| `endTime` | string | 是 | 结束时间，例如 `2026-12-31 23:59:59` |
| `status` | number | 是 | 模板默认填 `0` |
| `limitBrands` | array | 是 | 限定品牌数组，不限就填 `[]` |
| `desc` | string | 否 | 描述文案 |
| `issueMode` | string | 是 | `auto` 或 `code` |
| `redeemCode` | string | 条件必填 | 当 `issueMode = code` 时必填 |

### 4.1 字段规则补充

- `type = cash`
  - `value` 表示减多少钱，例如 `38`
- `type = discount`
  - `value` 表示折扣率，例如 `0.9` 表示 9 折
- `limitBrands = []`
  - 表示全场通用
- `limitBrands = ["Roja Dove"]`
  - 表示只对该品牌相关商品生效
- `issueMode = code` 但没填 `redeemCode`
  - 用户侧会提示“兑换码无效”
  - 控制台会输出配置警告
  - 这属于模板配置错误

## 8. 推荐测试流程

### 8.1 测自动发券

1. 在 `coupons` 集合中加入一张 `issueMode = auto` 的券
2. 清除开发者工具缓存
3. 重新启动小程序
4. 打开“我的”页和“优惠券”页

预期结果：

- 自动券已进入券包
- 我的页优惠券数量大于 0

### 8.2 测手动兑换

1. 在 `coupons` 集合中加入一张 `issueMode = code` 且配置了 `redeemCode` 的券
2. 打开优惠券页
3. 输入兑换码并点击“兑换”

预期结果：

- 提示“领取成功”
- 该券进入“可用券”列表

### 8.3 测配置错误

1. 在 `coupons` 集合中加入一张 `issueMode = code` 但不配置 `redeemCode` 的券
2. 在优惠券页尝试输入任何兑换码
3. 查看控制台日志

预期结果：

- 用户侧提示“兑换码无效”
- 控制台出现模板配置警告

### 8.4 测下单核销

1. 在购物车选择一张可用券
2. 完成结算
3. 重新进入优惠券页

预期结果：

- 该券进入“不可用”列表
- 状态显示为“已使用”

## 9. 数据修改后为什么前端没变化

如果你改了云端数据，但前端看起来没变化，按下面顺序检查：

### 9.1 先看缓存

当前项目有本地缓存，建议操作后：

1. 清除开发者工具缓存
2. 重新编译小程序
3. 重新进入页面

### 9.2 再看模板字段是否完整

尤其检查：

- `id` 是否唯一
- `issueMode` 是否正确填写
- `redeemCode` 是否在 `code` 券上配置
- 时间是否过期

### 9.3 再看控制台

重点关注：

- `云开发初始化成功`
- `自动发券同步失败`
- `coupon template misconfigured`

## 10. 常见问题

### Q1：为什么我新增了券，但用户没自动拿到？

可能原因：

- 这张券的 `issueMode` 不是 `auto`
- 模板字段没填完整
- 本地缓存没清
- 使用了旧模板 `id`

### Q2：为什么兑换码一直无效？

可能原因：

- 这张券不是 `code` 模式
- `redeemCode` 没配置
- 输入的兑换码和模板不一致
- 该券已经领过

### Q3：为什么删了模板，用户券还在？

这是当前设计行为：

- 用户已领到的券保留到过期
- 删除模板不会把用户本地已领券立刻删掉

### Q4：为什么同一个活动重新上线后，有些老用户没收到券？

因为当前版本中：

- 同一个模板 `id` 的自动券只会发一次
- 如果要再次发放，请新建模板 `id`

## 11. 建议的日常操作规范

### 发节日券

- 新建一张 `issueMode = auto` 的模板券
- 使用新的 `id`
- 发布后让用户重新进入小程序领取

### 发内部奖品券

- 新建一张 `issueMode = code` 的模板券
- 配置唯一的 `redeemCode`
- 把兑换码通过活动、客服或线下方式发给用户

### 做测试时

- 优先在测试环境或测试账号下操作
- 每次改完券模板后，记得清缓存再验证

## 12. 当前版本限制

当前版本已知限制：

- 不支持“一券多码”
- 不支持同模板周期性重复自动发放
- 用户券包仍然保存在本地，不在云端
- 没有后台运营页面，模板靠数据库手动维护

如果后续需要扩展，可考虑：

- 一券多码
- 发券批次
- 周期活动发券
- 云端用户券包
- 正式运营后台
